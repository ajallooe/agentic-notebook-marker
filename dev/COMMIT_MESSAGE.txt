Add --force-xargs debug option to test xargs fallback path

Added a debug option to force the use of xargs even when GNU parallel is
available. This ensures the xargs ARG_MAX fix can be tested properly.

## Problem

With GNU parallel now installed, there's no way to test that the xargs
fallback path works correctly with the new ARG_MAX fix. Need a way to force
xargs usage for testing purposes.

## Solution

Added `--force-xargs` flag that can be passed to marking scripts:

```bash
./mark_structured.sh assignments/lab1 --force-xargs
./mark_freeform.sh assignments/project1 --force-xargs
```

This flag:
- Skips GNU parallel even if available
- Forces use of xargs with the line-number approach
- Displays clear message: "Using xargs for task execution (forced via --force-xargs)"
- Enables testing xargs behavior with large task lists

## Implementation

**src/parallel_runner.sh:**
- Added `--force-xargs` flag to argument parser
- Modified parallel detection: `if command -v parallel && [[ $FORCE_XARGS == false ]]`
- Added conditional message for forced xargs mode
- Flag completely bypasses parallel, even if available

**mark_structured.sh:**
- Added argument parsing to accept `--force-xargs`
- Passes flag through to both parallel_runner.sh calls (markers and unifiers)
- Updated usage message to show new option

**mark_freeform.sh:**
- Same changes as mark_structured.sh
- Consistent interface across both marking workflows

## Usage Examples

**Normal execution (uses parallel if available):**
```bash
./mark_structured.sh assignments/lab1
```

**Force xargs for testing:**
```bash
./mark_structured.sh assignments/lab1 --force-xargs
```

**Output when forced:**
```
Parallel Task Runner
====================
Tasks file: /path/to/tasks.txt
Total tasks: 224
Concurrency: 4

Using xargs for task execution (forced via --force-xargs)
```

## Benefits

- **Testing**: Can verify xargs ARG_MAX fix works with large task lists
- **Debugging**: Helps isolate issues between parallel and xargs implementations
- **Verification**: Ensures fallback path remains functional
- **Development**: Useful for testing without uninstalling parallel

This is a development/testing feature that doesn't affect normal operations.
Users can continue using the default behavior (automatic parallel/xargs detection)
while developers can force specific execution paths for testing.
